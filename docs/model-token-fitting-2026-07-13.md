# 模型 Token 经验拟合记录（2026-07-13）

数据源：`usage_2026-06-14_to_2026-07-13.xlsx`

## 1. 已确认的产品共识

官方模型的输入、缓存写入、缓存读取和输出单价是公开价格，不需要拟合。

真正需要拟合的是：

> 在历史请求的输入、缓存写入、缓存读取和输出构成下，用户每花 1 单位站内余额，大约能得到多少 M Token。

因此，模型 Token 估算应直接由“实际到账站内余额 + 分段倍率 + 历史使用结构”得到。官方标价容量是独立的价格对照指标，不是用户 Token 估算必须经过的中间步骤。

真实人民币/美元汇率只参与“相对官方直购便宜多少”的比较，不参与本站 Token 数量预测。

## 2. 倍率归一化

工作簿中的实际记录包含两个倍率：

- 0.40：10,465条原始记录。
- 0.28：14,681条原始记录。

费用字段满足：

```text
用户扣费 ≈ 原始费用 × 当行倍率
```

为了利用全部真实使用样本，同时避免把0.28阶梯优惠误认为模型本身更便宜，每条记录统一折算到0.4基准：

```text
0.4等价扣费 = 用户扣费 × 0.4 / 当行倍率
```

由于用户扣费等于原始费用乘以当行倍率，上式等价于：

```text
0.4等价扣费 = 原始费用 × 0.4
```

## 3. 拟合公式

每条请求的总 Token：

```text
总 Token = 输入 + 输出 + 缓存读取 + 缓存创建
```

模型在0.4倍率下的经验系数：

```text
K_0.4 = 总 Token（M）之和 / 0.4等价扣费之和
```

单位为：

```text
M Token / 站内余额
```

任意倍率 m 下，每 1 站内余额预计得到：

```text
K_m = K_0.4 × 0.4 / m
```

阶梯计算必须逐段执行：

```text
预计总 Token（M）
= sum(该档站内余额 × K_0.4 × 0.4 / 该档倍率)
```

如果继续沿用当前前端的“标称容量 × 模型系数”内部实现，则兼容系数必须为：

```text
前端容量系数 = K_0.4 × 0.4
```

两种写法在数学上等价，但面向用户的解释应优先采用“站内余额 → 分段倍率 → Token”。

## 7. 结果解释

本批数据整体高度依赖缓存读取，尤其是Sol、GPT-5.5、GPT-5.4和Luna。高缓存命中会显著提高每单位余额得到的总 Token。

因此这些系数表示：

> 该用户在2026-06-14至2026-07-13的真实使用结构下，预计每单位站内余额能获得的总计费 Token。

它们不是模型固定价格，也不能保证低缓存、高输出用户得到相同结果。

生产页面应：

- 展示“预计值”，而不是承诺值。
- 明确总 Token 包含输入、输出、缓存读取和缓存创建。
- 对Mini、Luna和Spark保留明显的波动提示。
- 新增更多用户和更长时间样本后重新拟合，并保留本记录以便对比版本变化。
