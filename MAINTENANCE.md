# 项目维护规范

本文是 `qianchencc/calc` 的维护基线。修改计算逻辑、默认数据、部署配置或域名之前，必须先阅读并遵守本文。

## 1. 生产身份与唯一事实来源

- GitHub 仓库 `qianchencc/calc` 是代码、数据结构、初始化种子和灾备基线的唯一事实来源。动态控制面上线后，运行时模型注册表与生效拟合以Upstash Redis为事实来源，边界见ADR-0001。
- `main` 是生产分支；Vercel 从该分支自动构建并发布。
- 正式访问域名是 [calc.qianc.ltd](https://calc.qianc.ltd)。
- Vercel 项目为 `chen/calc`。
- 旧的 ChatGPT Sites 地址只作为历史验收记录，不得作为主站继续修改，也不得反向覆盖 GitHub 源码。
- 线上热修后必须把同一修改提交回 GitHub；禁止长期保留“线上和仓库不一致”的状态。

## 2. 技术结构

当前生产版本仍是Next.js App Router静态计算器。已接受的目标架构会增加服务端写入接口、Upstash Redis动态控制面和运行时密钥；在对应实现完成验证前，静态常量仍是线上实际数据来源。

关键文件：

- `app/page.tsx`：领域数据、计算函数、输入状态与页面结构。
- `app/globals.css`：完整视觉与响应式样式。
- `app/layout.tsx`：站点元数据。
- `public/favicon.svg`：站点图标。
- `.github/workflows/ci.yml`：类型检查与生产构建质量门。
- `vercel.json`：Vercel 构建配置。

不要把旧 Sites 项目的平台内部文件、临时预览文件或 Cloudflare Worker 代码复制进本仓库。

## 3. 计算口径：不得破坏的不变量

设：

- `P`：用户实际支付的人民币。
- `B`：充值后实际获得的站内余额，单位是“站内记账单位”，不是美元。
- `E`：真实美元兑人民币汇率，单位为人民币/美元。
- `m_i`：第 i 个计费档位的倍率。
- `B_i`：落入第 i 个档位的站内余额。
- `K_m,0.4`：模型 m 在0.4倍率真实使用结构下的经验系数，单位为 M Token/站内余额。

同样人民币直充官方可买到：

```text
Q_official = P / E
```

平台声明规则下的标称官方容量：

```text
Q_site = sum(B_i / m_i)
```

相对官方直购的容量倍数：

```text
F = Q_site / Q_official
```

每 1 官方标价美元容量的实际人民币成本：

```text
C = P / Q_site
```

模型 Token 估算不依赖真实汇率。先把历史实际扣费统一到0.4基准：

```text
charge_0.4 = 用户扣费 × 0.4 / 当行倍率
K_m,0.4 = sum(总Token_M) / sum(charge_0.4)
```

再对用户余额逐档计算：

```text
Token_m = sum(B_i × K_m,0.4 × 0.4 / m_i)
```

官方标价容量只用于价格对照。若当前前端内部仍使用 `Q_site × k_m`，兼容系数必须满足 `k_m = K_m,0.4 × 0.4`，不得重复应用倍率。

必须遵守：

1. 站内余额不是美元。即使界面显示 `$`，也只是站内记账符号。
2. “人民币 1:1 到账站内余额”描述的是 (B/P)，不能替代真实汇率 (E)。
3. 相对官方的总优惠由“人民币到账优势”和“模型倍率优势”共同形成，不能只算其中一层。
4. 阶梯必须分段计算，各段分别除以对应倍率；不得把全部余额套用最终档倍率。
5. 档位按当月累计实际扣除的站内余额判断；进入下一档后仅后续请求使用新倍率，不追溯重算此前扣费。
6. Token 总量口径为输入、输出、缓存读取、缓存创建之和。
7. 当前只展示参与付费估算的 GPT 模型；准确的 `-proxy` 路由后缀并入基础模型，免费模型必须排除。
8. 模型经验系数使用真实请求的混合结构；不同实际倍率的记录必须先归一化到同一基准，不能把阶梯优惠误认为模型价格变化。
9. 所有数字输入必须允许空字符串作为编辑态；空值参与计算时视为 0。禁止在 `onChange` 时立即把空值强制写回 `0`，否则会再次出现输入数字变成 `0x` 的问题。

## 4. 当前默认数据

### 4.1 真实汇率

- 默认值：1 美元 = 6.7989 元人民币。
- 日期：2026-07-10。
- 一手来源：[中国人民银行人民币汇率中间价公告](https://www.pbc.gov.cn/zhengcehuobisi/125207/125217/125925/2026071009003741941/index.html)。

汇率属于带日期的快照，不得在没有日期和来源的情况下写成“实时汇率”或“当前汇率”。

### 4.2 默认阶梯

| 当月累计实际扣费（站内余额） | 倍率 |
|---|---:|
| 0–30 | 0.40 |
| 30–70 | 0.32 |
| 70–130 | 0.28 |
| 130+ | 0.24 |

只有在站点所有者确认阶梯插件规则改变后才能修改。修改时必须同步更新默认值、界面说明、维护文档与回归案例。

### 4.3 当前生产拟合基线

以下是迁移前静态前端仍使用的0.4倍率基线，单位为“每 1 站内余额预计得到的 M Token”。动态控制面上线后，长期存储改为ADR-0002定义的倍率中性模型产出系数。

| 模型 | 0.4基准系数 |
|---|---:|
| GPT-5.6 Terra | 3.43466 |
| GPT-5.6 Sol | 3.16505 |
| GPT-5.6 Luna | 13.09546 |
| GPT-5.5 | 2.64776 |
| GPT-5.4 | 5.58383 |
| GPT-5.4 Mini | 4.63849 |

GPT-5.3 Codex Spark 是免费模型，不进入付费Token估算或模型选择器。完整拟合方法、阶梯换算与样本限制见 [模型 Token 经验拟合记录](./docs/model-token-fitting-2026-07-13.md)。

### 4.4 官方 Plus 对照基准

- 默认实际购买价格：¥150/月；高级模式允许用户自定义。
- 月度调用容量基准：由 Pro 20× 的每周实测容量 1700，除以20后再乘以 52/12 周折算。
- 页面必须同时显示“同预算容量性价比”和“相当于一份固定 Plus 月额度”两种口径，避免将不可拆分购买的 Plus 套餐误解为可按比例购买。
- Plus 对照属于经验基准，不参与官方 API 容量、站内余额或模型 Token 的计算。

## 5. 数据更新规范

### 更新真实汇率

- 只使用中国人民银行或中国外汇交易中心的一手公开数据。
- 在 `app/page.tsx` 中一次性更新汇率值、发布日期和原始来源链接。
- 汇率值与来源页面必须对应同一天。
- 修改后执行第 6 节全部必测案例。
- 若以后实现自动更新，必须显示数据时间、失败回退值和来源；不得静默使用抓取失败后的 0 或过期值。

### 更新阶梯规则

- 先取得站点所有者确认的插件规则或配置证据。
- 明确档位边界的归属、累计量的单位、是否跨月重置、是否追溯。
- 同步修改代码、界面解释、本文件和测试基准。
- 不允许仅根据一张不完整截图猜测生产规则。

### 更新模型系数

- 使用可审计的请求数据重新拟合，并记录样本时间范围、请求数、总 Token、实际用户扣费和每行倍率。
- 合并准确的路由后缀，排除非 GPT 数据和确认不应纳入的账户/异常样本。
- 单条请求或极小样本不得直接替换长期系数。
- 将每条实际扣费按 `用户扣费 × 基准倍率 / 当行倍率` 归一化后再聚合；禁止直接混合不同阶梯的用户扣费。
- 同时检查0.4基准系数、各阶梯换算、观察范围和界面说明；不能只替换表格中央值。
- Token 点估计使用聚合总 Token / 聚合归一化扣费，不能简单平均逐请求比率；真实使用中出现的输入、缓存和输出结构应进入经验结果，并在页面说明其样本属性。

## 6. 必测回归案例

任何涉及 `app/page.tsx`、计算函数、输入组件或默认数据的修改，至少验证：

| 场景 | 输入 | 预期 |
|---|---|---|
| 默认阶梯 | 支付 ¥100、到账 100、汇率 6.7989 | 标称容量 307.14；官方直购 14.71；相对官方 API 约 20.88× |
| 默认 Terra | 上述默认阶梯结果 | 预计约 421.97M Token |
| 官方 Plus 对照 | 上述默认阶梯、Plus 实付 ¥150/月 | 同预算容量性价比约 1.25×；相当于一份固定 Plus 月额度约 0.83× |
| 25 倍推演 | 支付 ¥10、到账 10、汇率 10、单倍率 0.4 | 官方直购 1；标称容量 25；相对官方 25× |
| 其他中转 | 支付 ¥100、到账 50、单倍率 0.5 | 标称容量 100 |
| 清空后输入 | 清空任一数字框，再输入 7 | 值必须为 7，不能变成 07、0x 或无法覆盖 |
| 阶梯边界 | 分别测试 30、70、130 及其相邻值 | 分段连续，无重复计费、漏计或追溯 |
| 非法参数 | 空值、0、负数、非递增档位 | 页面不崩溃，给出 0 或明确校验提示 |

同时检查桌面端与移动端，确认高级模式的字段说明、公式、来源链接和结果都可阅读。

## 7. 开发与发布流程

推荐流程：

1. 从最新 `main` 创建 `feature/*`、`fix/*` 或 `docs/*` 分支。
2. 在提交前运行：
   ```bash
   npm install
   npm run typecheck
   npm run build
   ```
3. 推送分支并使用 Vercel Preview 验证页面。
4. 通过 Pull Request 合并到 `main`；GitHub Actions 必须通过。
5. Vercel 自动生成 Production Deployment。
6. 验证 [calc.qianc.ltd](https://calc.qianc.ltd) 返回 HTTPS 200，并手测默认案例和本次受影响路径。

禁止对共享的 `main` 强制推送。依赖升级应独立提交，避免与计算规则调整混在一起。

文档提交也会触发 Vercel 构建；合并后仍需确认最新生产部署为 Ready。

## 8. 密钥与仓库卫生

禁止提交：

- GitHub、Vercel、Cloudflare API Token。
- SSH 密码、服务器密码或个人凭据。
- `.env*`、`.vercel/`、构建产物 `.next/`、依赖目录 `node_modules/`。
- 用户账单、原始请求日志或可识别个人身份的数据。
- 无法核验来源的汇率、价格或模型参数。

当前静态版本不需要运行时密钥。动态控制面将使用独立的拟合写入密钥和注册表管理密钥，全部存入Vercel Environment Variables；仓库和文档只记录变量名、用途与轮换方式，不记录值。

## 9. 域名与 Cloudflare 维护

- `calc.qianc.ltd` 应指向当前 Vercel 项目。
- DNS 调整只在明确的部署迁移或故障处理窗口进行。
- 更改记录类型、目标或 Cloudflare 代理状态前，先记录旧值与回滚方案。
- 启用 Cloudflare 代理前必须验证 HTTPS、缓存与 Vercel 域名校验不会受影响。
- 未准备回滚方案时，不得删除 Vercel 域名绑定或 Cloudflare DNS 记录。

## 10. 回滚与故障处理

- 常规回滚：在 GitHub revert 引入故障的提交，等待 Vercel 从 `main` 重新发布。
- 紧急回滚：可先在 Vercel 回滚到最近的健康部署，但随后必须立即在 GitHub 完成对应 revert，恢复源码与线上一致。
- Vercel 或域名临时故障时，可把旧 Sites 地址作为人工告知的应急入口；不得在两套源码上同时开发。
- 回滚后重新验证 HTTPS 200、默认 100 元案例、输入框清空重输和高级模式。

## 11. 发布记录最小字段

涉及计算或生产配置的发布，在 PR 或提交说明中记录：

- 发布日期与提交 SHA。
- 修改的数据来源或规则依据。
- 对计算结果的影响。
- `typecheck`、`build` 与必测案例结果。
- Preview/Production 验证结果。
- 对应的回滚提交或上一健康部署。
